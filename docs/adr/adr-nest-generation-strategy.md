# ADR: Nest generation strategy for `@anarchitects/nest`

## Status

Proposed

## Context

The `@anarchitects/nest` plugin is intended to provide a modern Nx-native integration layer for Nest applications and libraries.

The plugin must support the NestJS v12 direction while Nest v12 is still evolving. At the same time, it must follow Nx Project Crystal principles:

- infer targets through `createNodesV2`
- avoid unnecessary explicit `project.json` targets
- use command targets where possible
- declare explicit `inputs`, `outputs`, and `cache` behavior
- avoid custom executors unless programmatic control is genuinely required

Feedback from the NestJS side clarified that ecosystem tooling is free to choose the most practical generation mechanism. The Nest CLI is useful, but not mandatory. `@nestjs/schematics` remains a valid dependency for ecosystem tooling.

Feedback from the Nx side confirmed that the proposed direction aligns with Project Crystal and recommended starting with the core inference layer first: project detection, `createNodesV2`, inferred command targets, plugin options, cache metadata, and minimal e2e validation.

A scan of Nest v12 branches showed that:

- `@nestjs/schematics` v12 is the official generator package and exposes schematics such as `application`, `library`, `resource`, `controller`, `service`, `module`, `guard`, `interceptor`, `pipe`, `filter`, `middleware`, `gateway`, `resolver`, `class`, `interface`, and `decorator`.
- the v12 application schematic supports `type: esm | cjs`, defaults to ESM, and uses module-system-aware testing defaults: ESM projects use Vitest, CJS projects use Jest.
- the v12 Nest CLI delegates application and artifact generation to `@nestjs/schematics`.
- `nest new` also performs CLI-level side effects such as package installation, git initialization, `.gitignore` creation, console output, and process exit behavior.

These observations make direct schematic integration more appropriate for Nx generators than shelling out to the Nest CLI as the primary generation path.

## Decision

The `@anarchitects/nest` plugin will use the following generation strategy:

1. **Nx-native core first**

   The core plugin layer is fully Nx-native and independent of Nest CLI or schematics:

   - project detection
   - `createNodesV2`
   - target inference
   - workspace-level `pluginOptions`
   - project graph integration
   - explicit `inputs`, `outputs`, and `cache` metadata

2. **Schematics-first generation where useful**

   For Nest framework boilerplate, the plugin should prefer direct `@nestjs/schematics` integration over shelling out to the Nest CLI.

   The default application generator should use the `application` schematic as the source of Nest boilerplate and then apply Nx-specific metadata through Nx Tree transforms.

3. **Nest CLI as fallback and user-facing escape hatch**

   The Nest CLI remains useful and should be documented for standard artifact generation.

   Shelling out to `nest` from Nx generators should only be used when there is a concrete reason and when side effects are controlled through options such as `--dry-run`, `--skip-install`, and `--skip-git`.

4. **No thin wrappers for Nest artifacts**

   The plugin should not re-expose thin Nx wrappers for Nest artifacts such as:

   - controller
   - service
   - module
   - provider
   - guard
   - interceptor
   - pipe
   - filter
   - middleware
   - gateway
   - resolver
   - class, interface, decorator

   These should be generated directly through the Nest CLI or the underlying Nest schematics.

5. **Value-add generators only**

   Nx generators should exist where the plugin adds clear value beyond Nest itself:

   - `application`: Nx workspace integration around Nest application scaffolding
   - `library`: Nx-first library placement, metadata, tags, boundaries, and path aliases
   - `resource`: optional value-add wrapper around Nest resource generation for Nx path awareness, Fastify setup, schema-first validation, or architecture conventions

6. **No-template-drift policy**

   The plugin should avoid modifying Nest-generated files unless there is no additive alternative.

   Preferred operations:

   - create new files
   - add configuration files
   - augment JSON/JSONC configuration
   - add dependencies through Nx utilities
   - add Nx project metadata
   - add documentation and examples

   Discouraged operations:

   - raw string patching of generated Nest source files
   - rewriting `main.ts`, modules, controllers, or services by default
   - depending on exact Nest template contents
   - silently overwriting files generated by Nest schematics

   If a Nest-generated file must be modified, the change must be explicit, preferably opt-in, structure-aware or AST-aware, and covered by e2e tests.

7. **Additive transform guard**

   The generation adapter should provide a small helper to encode the no-template-drift policy in code.

   Proposed module:

   ```text
   packages/nest/src/generation-adapter/additive-transform.guard.ts
   ```

   Responsibilities:

   - create files only when missing unless overwrite is explicitly allowed
   - update JSON/JSONC config safely
   - add dependencies and Nx metadata through centralized helpers
   - require an explicit reason for destructive changes
   - warn or fail when modifying known Nest-generated source files without explicit override

8. **Nest v12 compatibility tracking**

   Because Nest v12 is still evolving, compatibility must be tracked explicitly through a dedicated tracking issue and CI compatibility tests.

   Supported upstream versions should be pinned and updated deliberately.

## Consequences

### Positive

- The plugin aligns with Nx Project Crystal.
- The plugin avoids unnecessary custom executors.
- The plugin avoids template drift by reusing official Nest schematics.
- The plugin keeps Nest as the source of framework conventions.
- The plugin keeps Nx responsible for workspace integration and target inference.
- The plugin can evolve with Nest v12 by tracking schematic and CLI changes explicitly.

### Negative / trade-offs

- Direct schematic integration requires ESM-compatible loading and adapter work.
- The plugin must handle Angular Devkit schematic execution inside an Nx generator context.
- Compatibility with evolving Nest v12 alpha/beta versions must be actively tracked.
- Avoiding file patches may make some optional enhancements more explicit or require additional files/configuration.

## Implementation notes

Initial implementation should proceed in this order:

1. Build the core Project Crystal layer:
   - package shell
   - project detection
   - `createNodesV2`
   - plugin options
   - inferred command targets
   - cache metadata
   - minimal e2e validation

2. Build a schematic adapter spike:
   - invoke `@nestjs/schematics:application`
   - test ESM app generation
   - test CJS app generation
   - verify dry-run-safe behavior
   - verify Nx Tree integration

3. Build the application generator:
   - use Nest application schematic
   - add Nx metadata
   - add dependencies via Nx utilities
   - support optional Fastify/SWC/schema enhancements without changing defaults

4. Build library/resource generators:
   - library generator remains Nx-first
   - resource generator is value-add only
   - no thin artifact wrappers
